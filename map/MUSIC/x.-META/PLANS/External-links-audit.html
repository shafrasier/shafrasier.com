External links audit

May 15, 202623 min read

This document tracks every named entity mentioned in MAP notes that (a) does not currently have its own MAP note and (b) needs an external link sourced. It is the single source of truth for “what’s been named in the MAP but doesn’t yet have a home or a link.” Every close-read of a new or revised note includes an External-link audit pass (checklist item 15 in x. META/Using the MAP.md) that updates this document.

The linking convention

  • Songs (including singles) → Apple Music. On first mention in prose: ["Song Title"](apple-music-url) (YEAR). Subsequent mentions: "Song Title" (quotes, no link). Exception: compositions without a canonical recording → Wikipedia, not Apple Music — Tin Pan Alley standards, jazz standards, hymns and spirituals, Stephen Foster songs, traditional folk material, show tunes, art songs, opera arias. The test: can you name THE recording most listeners would picture when the song is mentioned? If yes (the Crystals’ “Be My Baby,” Chuck Berry’s “Johnny B. Goode”) → Apple Music. If no (Gershwin’s “Summertime,” Berlin’s “White Christmas,” “Amazing Grace”) → Wikipedia.
  • Albums, artists, producers, songwriters, session musicians, labels, studios → Rate Your Music (RYM). On first mention: [*Album*](rym-url) (YEAR) for albums, [Name](rym-url) for everything else.
  • Executives and other figures outside RYM’s taxonomy → Wikipedia. On first mention: [Name](https://en.wikipedia.org/wiki/Name). Label founders and owners who never performed (Berry Gordy, Ahmet Ertegun, the Chess brothers), A&R figures (John Hammond, Ralph Peer), publishers (Don Kirshner), managers (Brian Epstein, Peter Grant), and similar non-performing industry figures get Wikipedia, not RYM. See the Wikipedia fallback rule in Using the MAP.md for the full list and reasoning.

Why the split. RYM has rich, contextual pages for every entity type except individual songs and non-performing industry figures. RYM song pages are typically thin and their sidebars frequently lack Apple Music streaming links, so sending a reader there forces a friction path (RYM song page → album page → AM icon); Apple Music gives immediate audio access. RYM does not meaningfully index executives, managers, and A&R people whose work is institutional rather than sonic, so Wikipedia is the richer destination for them. For everything else, RYM is the richer destination: album pages have streaming sidebars, genre tags, community ratings, and cross-links; artist/producer/songwriter pages catalog discography in a way no single streaming service does. The rule tracks where the target page is most useful, not where it’s most convenient in the moment.

Universal across note types. This convention applies to every note in the MAP — IDEAS notes, genre notes, scene notes, artist notes, album notes, everything. Every named, linkable entity gets its external link on first mention in every note it appears in, the first time each note is created or revised. The rule is a first-class linking principle on par with the wikilink conventions, not formatting minutiae.

When a MAP note is created for a previously externally-linked entity, every prior external link converts to the wikilink. External links are stopgaps. Once a note exists, the graph view and backlinks only work if the external links across the MAP are swept and replaced. When a new note lands, grep for the entity’s RYM, Apple Music, or Wikipedia URL across the vault, convert each external link to [[Note name]], and remove the entity from this document. The external-link audit pass (checklist item 15 in Using the MAP.md) must not skip externally-linked entities — they’re the highest-priority conversion candidates the moment the note ships.

Entries excluded because they already have their own MAP notes: any entity with a dedicated note in the vault — don’t add it here. If an entity has a note, it should be wikilinked on first prose mention instead. Songs already covered: “Will You Love Me Tomorrow,” “Be My Baby,” “Leader of the Pack,” “You Really Got Me,” “California Girls,” “Reach Out I’ll Be There,” “You Can’t Hurry Love,” “Strawberry Fields Forever,” “Penny Lane,” “Johnny B. Goode,” “Gee.”

Per-note audit tables — the default for new MAP notes (May 12, 2026 onward)

The dominant workflow going forward. When Claude writes a new MAP note, every external link the note introduces — Apple Music for songs, RYM for artists/albums/producers/songwriters/session musicians/labels, Wikipedia for studios/executives — goes into a single per-note table below, named after the note’s subject (e.g., an “Aretha Franklin” table for a new Aretha Franklin note). Columns: entity name, type, URL, location in note. Sha verifies all links in one focused pass, marks the table (done), Claude propagates them across the vault and removes the table.

This convention replaces the older topic-grouped tables (Songs by topic → Chicago blues canon, Chuck Berry catalog, etc.) for new-note work. Topic groupings were efficient for batch retroactive verification across multiple already-shipped notes, but they’re the wrong unit for incremental new-note work, where every link is fresh, the note’s links cohere as a single audit unit, and per-note tables make it trivial to handle catalog-section vs. body-prose placement separately (the “location” column tells propagation where each link belongs). Topic tables remain only for the retroactive backlog currently winding down — once that’s clear, all audit work happens through per-note tables.

Recently added external links — awaiting Sha’s verification

When Claude adds an external link during an edit to an existing note (not a full new-note creation), the link is logged in the legacy entity-type subsections below (Artists, Albums, etc.) until full transition to per-note tables. Sha verifies that the URL resolves to the correct page (the slug is right, the disambiguator is correct, the page isn’t a stub) and then deletes the row from this section. The verification is what closes the loop: without it, a wrong-slug link sits in a note indefinitely with no flag.
