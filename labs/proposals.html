1 · MAP / NYC — the geographic layer

Try first: Lab A — the Atlas (filters + year slider) and Lab B — the Time Machine (scrub the century, press play). Same ~36 hand-curated sites; two different theories of the page.

Why this belongs in the MAP

The MAP's whole thesis is that context is the most important thing in engaging with music. Place is context made literal. "Greenwich Village folk scene" already exists as a note — but the fact that the Gaslight, Café Wha?, Gerde's, and Dylan's first apartment sit within four blocks of each other is itself an argument about how scenes work, and only a map can make it. Geography also solves a real reader problem: it gives a non-linear, joyful way in for people who would never open an essay first.

The one architectural idea that matters

Places should be MAP notes, and the atlas should be a view — exactly as the graph is. Not a separate website bolted on.

Concretely: a venue like Studio 54 becomes a note (a new PLACES category, or INDUSTRY/VENUES) with frontmatter — coordinates, address, years active. A Quartz transformer collects every note carrying a location into a generated geo.json, and the atlas page renders whatever exists. The payoff compounds: a place note appears in the network graph (linked to its scenes, artists, records), on the atlas, in search, in backlinks. Write once in Obsidian, every view updates. No parallel database to maintain — that's the trap most "music map" projects die in.

Honest notes on usefulness (you asked)

Where this is genuinely strong: scenes (they're spatial by nature), the walking-tour reading on a phone, eras layered on the same block (52nd St jazz → Studio 54 within 600 feet), and the click-through into real notes — which is what no Google-My-Maps mashup has. Where it's weak: artists and records aren't inherently spatial; don't force every note onto the map. Rule of thumb — the atlas earns a pin wherever place explains something; it stays empty everywhere else. Precedents to be aware of (none integrated with a knowledge graph): one-off Google-Maps blog mashups of NYC venues, the Bowery-preservation history maps, Radio Garden (live radio by globe — great interaction, zero history).

Tech choice

Leaflet + CARTO's light basemap (what the labs use): ~42 KB, free with attribution, and the muted cartography already matches the MAP's cream. The upgrade path if you ever want fully custom map styling (cream land, taupe water, serif labels) is MapLibre GL + Protomaps — self-hostable single-file tiles, no API keys, no Google. I'd stay on Leaflet until the custom-cartography itch is real. Google's API: paid, branded, and wrong aesthetically — skip it.

2 · The scholarly architecture

Try first: The Library lab — your actual fifteen sources, each argument in one breath, and the "in conversation" rail that flips the shelf around an idea.

What you have (audit)

The source notes are genuinely good — consistent header (author/title/year/type), real summaries, "Key contributions," and "See also" links that already wire sources into ideas. What's missing is structure the site can use: the metadata lives in prose, so the Sources page can only be an alphabetical list; nothing can answer "show me the sociology," "what underwrites this idea," or "which notes cite this book." The x. prefix hack for sorting is a symptom of the same thing.

Is anyone else doing this? (you asked to be corrected)

You're essentially right. The closest things each lack half of yours: Grove/Oxford Music Online is authoritative but paywalled, unopinionated, and has no canon or listening layer. RYM has data with no argument; Pitchfork's Sunday Review has canon with no citations; Genius has annotation with no scholarship; Wikipedia has citations with no point of view (and proportion problems you've noted); 33⅓ does one album per book, offline; Are.na has the connective structure with no domain knowledge. A personal canon + scholarly apparatus + streaming links + a knowledge graph, in public — that combination is genuinely unoccupied. The defensible phrase is something like "a public musicology layer over a personal canon."

The four pieces I'd build (in order)

(1) Source frontmatter — the schema in the Library lab's footer: type: source, author, year, kind, discipline, informs: (wikilinks to ideas), goodreads/worldcat. One vault pass; I can draft it for your review since the vault isn't git-tracked.

(2) "Cited in" — automatic. A Quartz transformer indexes every wikilink pointing at a source note and emits two things: a "Cited in 12 notes" line on the source page, and a generated Sources drawn on block at the foot of every note that links sources. Zero maintenance; the prose you already write becomes the citation graph.

(3) The Library page — the lab, productionized as the SOURCES landing: argument-first cards, discipline/kind filters, the conversation rail, "start here" shelves (five books to understand the color line, three to understand authenticity…).

(4) Claims with page numbers (the big one, phased). House style: load-bearing factual claims get an Obsidian footnote naming source + page — [^1]: [[Wald — Escaping the Delta]], pp. 105–122. Quartz renders footnotes already; the transformer can then count, link, and eventually audit them (a build report of claims-without-citations). That's the full answer to the AI-writing worry: not a disclaimer, an apparatus — arguments that show their work, checkable by anyone.

Two reader-facing graces once (1)–(3) exist: every source gets a WorldCat / Open Library link next to Goodreads (find it in a physical library — right gesture for this project), and ideas notes get a small "The shelf behind this" footer generated from informs:.

3 · The streaming toggle — shipped tonight

Already live on the real site (it rode the night's fix batch): a small ♪ Apple Music / Spotify pill next to dark-mode (desktop sidebar + the home drawer). Flipping it rewrites every prose song link to a Spotify search for the title — no per-link manual work, never touches the explicit "Apple Music" metadata links on release pages, remembers the choice, flips back losslessly. The honest limitation: a search lands you one tap from the song rather than on it. The clean upgrade later is Odesli/song.link — feed it the Apple URL you already have and it resolves the exact track on every service; a build-time pass could bake verified Spotify IDs in with no ongoing effort from you.
